          Notification Mode Guide
        

        
      

      

        

          
What Mode Controls

          
Mode controls which notification importance levels may speak through the notifier policy. It does not change the event severity field. Severity remains separate and still means debug, information, warning, or error.

          
Importance is ordered from quietest to loudest intent: low_importance, neutral, urgent1, urgent2, danger1, danger2.

        

        

          
Precedence

          
    
            
  1. Manual DND 1 or Manual DND 2 wins while its timeout is active.
    2. 
            
  2. Enabled schedules win next. Schedule 2 wins over Schedule 1 when both are active.
    3. 
            
  3. The top Mode selector is the base mode used when no manual timeout or active schedule overrides it.
    4. 
          

          
Choosing Default in the top Mode selector does not disable enabled schedules. The schedules will apply again at their next active window.

Volumes And Speech Surfaces

          
Normal volume is used by Default. Quiet volume is used by scheduled and manual DND modes. Debug mode uses the stored debug volume. Phone listeners always speak when policy allows TTS. One desktop webpage speaks limits desktop browser speech to one active page per client IP for each event.

        

        

          
Danger2 Alarm

          
Danger2 is always armed in policy. The selected alarm sound can be picked and tested here, and its test playback uses the separate alarm volume. Automatic Danger2 alarm playback is still reserved for a safety-tested notifier path, so the current Pick and Test controls are for choosing and verifying the sound without creating accidental alarms.

How the protection paths fit together


        

          What this page is measuring
          

            
The status cards above come from the live Blueprints endpoint /api/v1/litellm/guardrail-status, which reads the node-local LiteLLM configuration at /xarta-node/.lone-wolf/stacks/litellm/config.yaml.

            
The test runner posts to /api/v1/litellm/injection-test, which then exercises the local LiteLLM chat path using the PRIMARY-LOCAL model alias and a canary token.

          

        


        

          Safe Web Research skill — separate first safety layer
          

            
The research workflow lives at /root/xarta-node/.xarta/.claude/skills/safe-web-research/. Its verified helper scripts include searxng-search.sh, scrapling-fetch.sh, crawl4ai-crawl.sh, liteparse-url.sh, and playwright-probe.sh.

            
Those scripts pass content through sanitize-web-output.py, which strips suspicious instruction lines, secret-exfiltration requests, install or SSH instructions, destructive commands, large blobs or data URIs, duplicate spam, and truncates oversized output.

          

        


        

          When one request hits one layer vs both layers
          

            
    
              
  1. If a request only searches or fetches web content locally, the Safe Web Research skill is the relevant first safety layer.
    2. 
              
  2. If that fetched content is then sent through the node-local LiteLLM gateway for summarising or reasoning, the LiteLLM guardrails shown above can also apply on the model path.
    3. 
              
  3. Direct MCP tool calls use LiteLLM’s /mcp/ JSON-RPC path, which is separate from /v1/chat/completions. The tool call itself is not the same thing as a model generation step.
    4.

      WARN
      Integrity Check
    

    
  

  

    

      This node's database has been marked as degraded. Sync is paused — no outgoing
      writes will be sent to peers, and this node will not accept incoming sync actions until recovery.
    

    

      Common cause: A Force Restore was applied (disaster recovery). The restored
      database is flagged as degraded until a healthy peer pushes its full backup to this node.
    

    
Diagnostics

    

       Loading…
    

    

      
How to recover

      
    
        
  1. Wait — on every boot the app requests a full backup from the first reachable peer.
            If peers are online, recovery is automatic within ~10–30 s of (re)starting the service.
    2. 
        
  2. Restart the service — if the node has been running for a while without recovering,
            restarting triggers boot-catchup again:
            
            systemctl restart blueprints-app
    3. 
        
  3. Manual restore — navigate to Settings › Backups and restore from a
            known-good backup, then restart the service.
    4. 
        
  4. No peers available? — if this is the only node, you can reset integrity manually
            from the SQLite shell:
            
            UPDATE sync_meta SET value='true' WHERE key='integrity_ok';
            Only do this if you are certain the data is uncorrupted.
